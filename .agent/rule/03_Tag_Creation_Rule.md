# 🏷️ Rule: 태그 생성 및 명명 규칙 (Tag Creation Rule)
**Precedence: 1**

## 1. 보수적 생성 및 택소노미 기반 검증 (Taxonomy-First)
- 태그는 단순한 텍스트 쪼가리가 아니라 다차원 분류를 위한 네임스페이스 계층입니다.
- 신규 태그를 자동 생성하기 전에 반드시 `04_Tag_Taxonomy_Rule.md`를 조회하여, 기존 15개 네임스페이스 하위에 동의어나 유사 분류가 있는지 파악해야 합니다. (기존 값 우선 사용 원칙)

## 2. 신규 태그 명명 규칙 (Naming Convention)
- **소문자 케밥 케이스(kebab-case) 강제**: 띄어쓰기나 언더바(`_`)를 절대 사용하지 않으며, 모든 영문은 소문자, 단어 사이는 하이픈(`-`)으로 연결합니다. (예: `performance-marketing`, `air-purifier`)
- **계층형 중첩(`/`) 준수**: 모든 신규 태그는 반드시 지정된 15개 네임스페이스 중 하나에 소속되어야 합니다. (예: `#topic/tech/new-tech`)
- **고유 개체 태그화 금지**: 특정 거래처명, 인물명, 특정 제품 모델명은 태그로 생성하지 말고, 위키링크(`[[ ]]`)를 사용해 개체(Entity)로 취급하십시오.

## 3. 오토-업데이트 원칙 (Auto-Updating Dictionary)
- 사전에 없는 완전히 새로운 핵심 태그를 불가피하게 생성했다면, 에이전트는 즉시 문서 편집 도구를 활용하여 **`04_Tag_Taxonomy_Rule.md` 파일 내 해당 네임스페이스 리스트에 신규 값을 영구 등록(업데이트)**해야 합니다.
- 이 과정을 통해 택소노미 사전이 사용자 업무와 함께 실시간으로 진화하도록 보장합니다.

## 4. 엄격한 속성 통제 (Strict Property Control)
- `category` (문서 유형), `domain` (업무 영역), `status` (문서 상태) 프론트매터 값은 시스템 운영을 위한 근간 카테고리입니다.
- 에이전트는 사용자의 명시적 지시(승인) 없이는 이 3가지 메타데이터 속성의 새로운 값을 스스로 창조해서는 안 됩니다. 오직 사전에 정의된 규칙 내의 값만 매핑해야 합니다.
