# 📋 Rule: PPT 정밀 파싱 및 구조화 세부 규칙
**적용 대상**: PPT_Editor_Agent, PPT_Writer_Agent

## 1. 변환 최우선순위 (Preservation Hierarchy)
변환 시 다음 순서로 우선순위를 둡니다. 아래 단계가 손실될 때는 상위 단계로 보상해야 합니다 (예: 색으로 구분된 카테고리를 잃었다면 텍스트 라벨로 명시).
1. **의미(Semantics)** — 슬라이드가 전달하려는 핵심 메시지
2. **구조(Structure)** — 계층, 순서, 관계
3. **데이터(Data)** — 표·수치·라벨의 정확성
4. **시각(Visual)** — 도식·색상·배치

## 2. 슬라이드 계층 및 타이포그래피 정밀 매핑 (Hierarchy & Typography)
LG 보고서 디자인 시스템(v3.4+)의 고정 폰트 크기와 볼드 속성을 역산하여 마크다운 계층으로 100% 매핑합니다.
**[필수 구조 유지 지침]**
모든 슬라이드는 반드시 아래의 마크다운 헤더 구조를 최상단에 유지해야 합니다:
`# [슬라이드 번호] Page - [슬라이드 제목]`
`## [거버넌스 문장 (핵심 메시지)]`

- **[오염 요소(Noise) 텍스트 강력 삭제 의무]**: 슬라이드 상단의 제목을 구분하는 직선바(가로선) 위쪽에 위치한 텍스트 중, **진짜 슬라이드 제목(타이틀)을 제외한 모든 글로벌 네비게이션 요소(예: `1. 변화`, `2. 현황`, `3. 실행` 등의 탭/목차)** 및 문서 하단의 페이지 번호(예: `- 8/12`)는 절대적으로 불필요한 노이즈입니다. 구조화 단계에서 무조건 일괄 삭제(소거)해야 합니다.

- **20pt Bold (Navigation Title)**: 슬라이드 좌상단(I. 수익성 현황 진단 등) ➡️ 마크다운 `# ` (H1) 및 제목으로 병합.
- **Governance Message (거버넌스 메시지)**: 슬라이드 최상단의 '제목(Title)' 바로 밑에 위치하는 긴 텍스트(주로 직선바 아래에 위치)는 폰트 크기와 무관하게 무조건 거버넌스 메시지로 취급하여 마크다운 `## ` H2로 변환합니다. **(주의: 거버넌스 문장은 여러 줄일 수 있으므로 전체를 묶어 H2로 처리하십시오.) 단, 슬라이드 제목에 `[유첨]` 등 부록임을 나타내는 키워드가 포함된 경우 거버넌스 문장이 생략되었을 가능성이 큽니다. 이때는 억지로 본문 텍스트나 표의 헤더를 거버넌스로 오인하여 H2로 끌어올리지 마십시오.**
- **Sub-Title / Phase Header (소제목)**: 폰트(14pt Bold 등) 단서 외에도 다음과 같은 **시각적 형태**를 띠는 텍스트는 반드시 소제목(마크다운 `#### ` H4)으로 식별해야 합니다:
  1) 텍스트 바로 하단에 선(가로선)이 그어져 영역을 구분하는 경우
  2) 텍스트 바로 앞에 붉은색 등의 작은 가로바(장식)가 붙어있는 경우
  3) 직사각형 형태의 색상 박스 안에 텍스트가 단독으로 채워진 경우
  4) **다단/분할 레이아웃의 상단에서, 해당 구역 너비만큼 꽉 차는 가로로 긴 도형 안에 텍스트가 (주로 가운데) 정렬되어 들어간 경우** (이 형태는 무조건 해당 구역의 최상위 소제목입니다)
- **본문 하단 실선바 아래 텍스트 (Global Footnote)**: 슬라이드 하단 영역을 마무리하는 긴 가로 실선(Line)이 있고 그 밑에 텍스트나 URL이 배치되어 있다면, 이는 특정 요소에 종속된 것이 아니라 **해당 슬라이드 전체에 대한 범용 각주(참고자료)** 입니다. 반드시 최하단에 `> *※ 참고/각주: ...*` 형태(또는 URL의 경우 `> *※ 참고 URL: ...*`)로 분리하여 렌더링해야 합니다.
- **12pt Regular (Body Text)**: 본문 텍스트, 설명, 디테일 셀 ➡️ 일반 마크다운 불릿(`-`, `*`) 및 텍스트.
- **10pt Regular (Sub-text) 및 녹색 텍스트(#006600 등)**: `※` 같은 특정 기호가 없더라도, `#006600` 또는 그와 유사한 **녹색 계열의 폰트가 사용된 모든 글씨는 무조건 풋노트(Footnote)로 취급**해야 합니다. 단순 텍스트로 분리하여 고립시키지 말고, **반드시 관련된 시각적 인접 본문(바로 위 문장이나 리스트)의 하위에 종속시켜 결합**해야 합니다 (예: 들여쓰기된 이탤릭체 `    - *※ 풋노트 내용*` 또는 인용구 구조 사용).
- **고립된 수치/예산 풋노트 매핑 (Orphaned Budget/Resource Footnotes)**: 슬라이드 하단에 예산(예: `1.5 억`), 투입 리소스(예: `6MM`), 기간 등이 포함된 텍스트 조각들이 맵핑 없이 버려져 있다면, 이는 상단의 카드/프로세스(예: `1. 서포트 대응`, `2. FAQ 허브` 등) 각각에 속한 속성값입니다. 절대 하단에 방치하지 말고, 내용의 맥락이나 순서를 추론하여 **각 해당 항목의 텍스트 바로 밑에 `> *※ 리소스/예산: ...*` 형태로 완벽하게 1:1 매핑 및 종속**시켜야 합니다.
- **표(Table) 및 차트(Graph/Chart) 주변 텍스트 결합 규칙**: 표나 차트 주변(또는 내부)에 위치한 텍스트들은 절대 산발적인 리스트로 고립시키지 마십시오. 위치에 따라 다음과 같이 표/차트와 완벽하게 종속시켜야 합니다:
  - **상단(위)** 텍스트: **'제목(Title)'** (예: `#### 표/차트 제목`)
  - **우상단** 텍스트(작은 글씨): **'범례(Legend)'** (예: `> *※ 범례: ...*` 형태로 상단 또는 하단에 배치)
  - **하단(아래)** 텍스트: **'참고/출처(Reference)'** (예: `> *※ 참고: ...*` 또는 `> *※ Source: ...*` 형태로 하단에 배치)
  - **내부(Inside)** 텍스트 (차트 한정): 그래프 내부에 적힌 글씨는 해당 그래프의 데이터 설명이나 인사이트이므로, **차트 데이터 정리 블록 바로 하단**에 `> *※ 내부 설명: ...*` 형태로 결합하여 기재합니다.
- **12pt Bold (Inline Emphasis)**: 본문 내 핵심 어휘 강조 ➡️ 마크다운 `**강조**` 텍스트로 파싱.

## 3. 시각적 단서(Visual Cues)의 시맨틱 매핑
PPT의 컬러 규칙을 논리적 태그로 변환합니다.
- **Active Red (`#C00000`)**: 강조/위험/부정적 데이터. ➡️ `==Highlight==` 나 `**[CRITICAL]**` 태그 추가.
- **Bright Green (`#008000`)**: 보충 설명, 긍정 방향, 행동(Action) 칩. ➡️ `[보충]`, `[긍정]` 텍스트 라벨 추가.
- **Light Pink (`#FFF3F3`)**: 2~3안 비교 시 Cons(단점) 행. ➡️ 테이블이나 리스트 변환 시 명시적으로 `**[단점]**` 레이블 삽입.
- **Dark Grey (`#595959`) vs Black (`#000000`)**: 차트나 표에서 Black이 포커스(자사/목표)이고 Grey가 베이스라인(경쟁사)인 경우, 텍스트 서술 시 이를 구분하여 명시.

## 4. 표준 다이어그램 16종 다단계 폴백 (LG 16 Standard Diagrams)
LG 보고서의 16개 표준 다이어그램 패턴을 감지하여 다음과 같이 구조화된 마크다운으로 치환합니다.

**Level 1 (도식 유형별 Markdown 완벽 매핑)**
- **Issue Tree (원인 분해)**: 논리 흐름을 들여쓰기가 적용된 중첩 리스트(Nested List)로 100% 치환.
- **Process / Chevron Flow (순차 프로세스)**: 번호 매기기(`1. 2. 3.`)나 헤더(`#### Phase N:`)를 사용하여 순차 나열.
- **Numbered Card Stack (①②③ 카드)**: 각 카드를 `### ① 제목` 구조로 나누어 병렬 구조 유지.
- **다단/분할 레이아웃 (Split Layout)**: 한 페이지가 좌/우 2개 이상의 독립된 영역으로 나뉘고 각각 소제목을 가진 경우, 텍스트 덤프 시 위아래로 무작위로 섞지 마십시오. 반드시 '왼쪽 영역 소제목(`####`)과 그 하위 단락 1개', '오른쪽 영역 소제목(`####`)과 그 하위 단락 1개' 형태로 구역을 명확히 분리하고, 내용을 묶어(Paragraph) 구조화해야 합니다.
  - **[예외 형태] 좌우 분할 (일정 vs 효과)**: 한 슬라이드 내에 좌측은 '개발 일정(가로 텍스트)', 우측은 '기대 효과' 등으로 명확히 분할된 레이아웃일 경우, 각 영역을 독립적인 시맨틱 블록(`### 개발 일정` / `### 기대 효과`)으로 완벽히 구획화하여 정리해야 합니다.
- **[예외 형태] 상단 주요 메시지 / 하단 예시 도식**: 상단에는 프로세스 등 주요 메시지가 배치되고 하단에는 이를 뒷받침하는 도식/표가 나열된 경우, 하단의 표/도식은 단순 나열하지 말고 상단 메시지의 하위 항목인 **예시(Example) 블록**으로 종속시켜 문맥을 연결해야 합니다. (예시 데이터가 없더라도 구조상 예시로 인식해야 함)
- **[예외 형태] AS-IS / TO-BE 비교 구문**: 소제목이나 화면 상단에 AS-IS와 TO-BE가 배치되어 있는 경우, 두 상태를 표나 대조되는 리스트 형태로 명확하게 구분지어야 합니다. 특히 **TO-BE 영역의 프로세스형 도식은 전체 흐름을 완벽하게 이해하고 시맨틱하게(Mermaid 또는 순차 리스트) 재구성**해야 합니다.
- **Grid / Area Box (바둑판형 영역 박스 도식)**: 하나의 큰 주제 아래 여러 개의 독립된 영역 박스가 바둑판(Grid) 형태로 나열된 경우, 텍스트의 단순 평면적 나열을 절대 금지합니다. 각 박스의 '주제(Theme)'를 소제목(`###` 또는 `- **[주제]**`)으로 삼고, 그 아래 세부 내용과 수행 주체(Owner)를 매핑하여 시맨틱한 '단락(Block)' 단위로 완벽하게 재조립해야 합니다.
- **매트릭스/비교형**: Markdown Table(표)로 완벽 변환. 열과 행의 라벨(12pt Bold 등)을 표의 헤더로 정확히 배치.
- **Timeline / WBS / 로드맵**: **(강제 룰)** 텍스트의 형태가 어떠하든 표나 리스트 상단 헤더에 '월(Month)', '분기(Quarter)', '반기(1H/2H)', '연도(Year)' 등을 나타내는 시계열 데이터(예: 1월, Q1, 2026. 1H, 2027 등)가 순차적으로 나열되어 있다면, 이는 무조건 중장기 로드맵 또는 WBS(일정표)입니다. 이런 슬라이드는 절대 텍스트나 불릿 리스트로 단순 나열해서는 안 되며, **무조건 마크다운 표(Table)** 로 변환해야 합니다. (`| 수행 주체(구분) | Task (업무 내용) | 일정 (시작-종료) |` 또는 타임라인 형태의 시맨틱한 표로 재구성)
- **도형 기반 표(Table) 변환 규칙**: 헤더 열이나 행 역할을 하는 기준 도형이 존재하고, 그 도형의 너비나 높이와 일치하는 여러 도형들이 규칙적으로 나열된 다단/그리드(Grid) 형태라면, 이를 단순 텍스트로 흩뿌리지 말고 **반드시 마크다운 표(Table)** 로 완벽하게 재구성하여 해석해야 합니다.
  - **[특수 형태] 선이 없는 그리드 표 (Invisible Grid Table)**: 상단에 도형(예: HS, MS, ES, VS)이 나란히 배치되어 있고, 하단에 텍스트 블록들이 일정한 간격과 통일성을 가지고 바둑판식으로 배열된 경우, 그리드 선(가로/세로선)이 보이지 않더라도 이는 단순 다단/소제목이 아니라 **하나의 완벽한 표(Table)**입니다. 상단 도형들을 열 헤더(Column Header)로 삼고, 좌측 측면 등에 위치한 라벨(예: Comm 활동 목표, 우선 과제 등)을 행(Row)으로 삼아 마크다운 표로 묶어내야 합니다. 또한, 상단 첫 도형을 거버넌스로 오인하지 마십시오.
  - **[특수 형태] 중앙 헤더 공유 프로세스 표 (Middle-Header Process Table)**: 화면 중앙을 가로지르는 화살표나 도형이 나열되어 위/아래 영역을 분할하고, 상하단에 메시지가 배치된 도식입니다. 시각적으로 '핵심 목표', 'Key Message' 같은 라벨이 중앙 도형 밖(상단이나 측면)에 떠 있더라도, 이는 장식이 아니라 **표의 열(Column)을 정의하는 진짜 헤더**입니다. 따라서 상단 주제(예: Smart Factory)와 하단 주제(예: Robot)를 각각의 행(Row)으로 삼고, 외부에 떠 있는 라벨들을 열 헤더(Column Header)로 삼아 **표의 셀 안으로 데이터를 완벽하게 매핑**하여 하나의 거대한 마크다운 표로 꽉 채워 넣어야 합니다. 내용을 비워두거나 표 밖으로 흩뿌리지 마십시오.
  - **[특수 형태] 표 위 부유 주석 (Floating Annotations)**: 표(Table) 위나 옆에 말풍선, 뱃지, 별도 도형 형태로 떠 있는 주석 텍스트(예: "개발중", "New", "개선" 등)가 추출된 경우, 이를 표 바깥에 파편화된 리스트로 방치하지 마십시오. 해당 주석이 가리키는 표 내부의 특정 셀(Cell)이나 영역을 문맥상 파악하여, **표 내부 해당 데이터 뒤에 괄호 `(주석내용)` 형태로 완벽하게 편입(Mapping)** 시켜야 합니다.
- **단일 계열 도넛 차트(Doughnut Chart) 간소화**: 차트 위에 제목 텍스트가 있고, 도넛 차트 중앙에 단일 숫자(퍼센트 등)가 있으며, 하단에 각주가 달린 단순 도넛 차트는 복잡한 표(`| Series |...|`)로 역산하지 마십시오. 지표의 가독성을 위해 **`- [텍스트] - 0% - ([각주텍스트])`** 형태의 한 줄짜리 인라인 텍스트로 간소화하여 부모 항목에 종속시켜야 합니다.
- **결론/기대효과 (Bottom-wide Shapes)**: 슬라이드 하단에 전체 컨텐츠 너비만큼 꽉 차는 넓은 도형이나 텍스트 블록이 배치되어 있다면, 이는 거버넌스 메시지가 아니라 해당 도식의 **'결론(Conclusion) 또는 시사점'**을 의미합니다. 슬라이드 내용 최하단에 `> *※ 결론/시사점: ...*` 형태로 강조하여 명시하십시오.
- **분할선/구분선 (Section Divider Lines) 및 WBS 공유 헤더 유지**: 컨텐츠 가운데를 가로지르는 긴 가로선/세로선이 있을 때, 해당 선이 **표(Table)나 WBS(로드맵)의 하위 행(Row)들을 시각적으로 구분하기 위한 용도**인지, 아니면 완전히 다른 도식으로 전환되는 **섹션 분리선**인지 반드시 맥락을 살펴야 합니다. 특히 상단에 '시계열 데이터(연도, 분기 등)'가 헤더로 존재한다면, 그 아래에 그어진 가로선은 섹션 분리선이 아닙니다. **하단의 모든 내용(지표, 세부 과제 등) 역시 상단 시계열 헤더를 공유하는 동일한 WBS 표의 연속된 하위 행(Row)들로 해석**하여, 절대 헤더를 날리지 말고 하나의 거대한 마크다운 표로 통합해야 합니다.

**Level 2 (데이터 및 구조 서술식 차트 폴백)**
- **Mermaid 남용 금지**: 화살표(➡️)가 있거나 명확한 순차적 프로세스 형태의 도식이 아닌 한, 파편화된 단어 뭉치나 표(Matrix) 데이터를 무리하게 Mermaid 다이어그램(Flowchart)으로 변환하는 것을 절대 금지합니다.
- **배경/맥락(Context) 슬라이드**: 전체 논리의 배경을 설명하는 슬라이드는 단순 텍스트 나열을 지양하고, `거버넌스 메시지 ➡️ 소제목 ➡️ 핵심 문구` 순으로 논리를 부드럽게 연결하는 서술형 문단(Paragraph) 구조로 재작성해야 합니다. 예시적인 시각 자료(그래프 파편 등)는 압축적으로 요약만 전달합니다.
- **복잡한 개념 도식(Conceptual Diagram)**: Mermaid로도 소화할 수 없고 표로도 만들기 난해한 고도화된 개념 도식은 파편화된 텍스트 나열을 절대 금지합니다. 전체 맥락을 분석한 뒤, 각 덩어리의 핵심 메시지를 추출하여 "논리적 문단(Paragraph) 형태"로 완벽하게 재배치/문장화해야 합니다.
- Mermaid 적용이 적합하지 않은 데이터는 상기 규칙에 따라 문단으로 구성한 뒤, 필요시 `> [!tip] 🤖 AI 해석:` 블록을 통해 전체 맥락(Context)을 추가로 요약 서술합니다.

## 5. 부가 요소 및 고유 컴포넌트 (LG Components)
- **Big Number Box / KPI Bar**: 슬라이드의 핵심 수치가 강조된 박스/타일은 요약 섹션(`> [!info] KPI: [항목명] 수치`)으로 상단에 분리.
- **Insight Box (핵심 시사점)**: 하단 시사점 박스는 `> [!NOTE] 핵심 시사점` 블록으로 변환.
- **발표자 노트**: 슬라이드 본문 하단에 `> **발표자 노트**:` 형태의 인용 블록으로 별도 추가.
- **AI 해석 (AI Interpretation) 의무**: 단순히 파편화된 단어만 나열되어 있거나 표/다이어그램의 시각적 해석이 난해한 슬라이드의 경우, Writer Agent가 앞뒤 문맥을 종합하여 해당 슬라이드 본문 하단에 `> [!tip] 🤖 AI 해석: [종합된 문장형 요약 및 인사이트]` 형태의 콜아웃 박스를 의무적으로 추가해야 합니다.

## 6. 변환 품질 평가 기준 (Quality Criteria - Reviewer 전용)
작업 완료 후 다음 4가지 질문에 "예"라고 답할 수 있어야 합니다:
1. **재구성 가능성**: 마크다운만 보고 원본 PPT의 메시지 흐름을 재구성할 수 있는가?
2. **검색 가능성**: 키워드로 원하는 내용을 텍스트 검색할 수 있는가?
3. **구조 보존성**: 계층과 관계가 유지되었는가?
4. **읽기 흐름**: 별도 안내 없이 위→아래로 자연스럽게 읽히는가?

## 7. 링크 및 메타데이터 후처리 지침
- **메타데이터 및 링크 매핑**: 최상단 원본 파일 하이퍼링크 매핑(`👉 [원본 파일 열기]`) 및 문서 하단 연관 문서 추천(`> [!info] 🔗 자동 추천 연관 문서`)을 추가합니다.
- **이미지 링크 오류 방지 (URL Encoding)**: 이미지 파일 경로에 공백이나 대괄호(`[` , `]`)가 포함되어 있을 경우 마크다운 이미지 렌더링이 깨지므로, 반드시 모든 이미지 경로는 URL 인코딩(공백은 `%20`, 괄호는 `%5B`, `%5D`) 처리하여 완벽하게 렌더링되도록 조치해야 합니다.
- **Excel/CSV 원본 삭제 방침**: 가치가 떨어지는 순수 덤프 데이터 발견 시 즉시 삭제 처리.

## 8. 예외 처리 지침 (UTF-8 Stream Error 대응)
- **증상**: 대용량 마크다운 텍스트나 복잡한 유니코드(한글, 이모지, 보이지 않는 공백 등)가 포함된 문서를 일괄 변환(`replace_file_content`)할 때 `field contains invalid UTF-8`와 같은 Stream Error 발생.
- **대응 방안**: 
  1. 에러 발생 시, 해당 수정 블록을 포기하지 말고 문제 원인을 파악합니다.
  2. 대규모 청크 수정을 지양하고, Python 스크립트(`run_command` via `python -c`)를 활용한 정밀한 라인 교체(Injection) 방식으로 우회하여 변환 충돌을 해결해야 합니다.
