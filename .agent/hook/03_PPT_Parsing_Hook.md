# 🚫 Hook: PPT 파싱 보호 조치 절대 규칙

## 1. 세션당 파일 처리 수량 제한 (Max Files Limit)
- **절대 규칙**: LLM 토큰 한계 초과와 컨텍스트 누락 방지를 위해, **어떤 상황에서도 한 턴(세션)당 최대 2개의 문서까지만** 처리해야 합니다.
- **대응 방안**: 사용자가 3개 이상의 파일을 한 번에 요청하더라도, 에이전트는 무조건 "토큰 한계로 인해 2개씩 분할 처리하겠습니다"라고 고지한 뒤 앞의 2개만 먼저 진행해야 합니다.

## 2. 불완전 종료 금지 (No Incomplete Workflow)
- **절대 규칙**: 파이썬 스크립트(Parser) 구동이 완료되었다고 해서 사용자에게 "변환이 끝났다"고 보고해서는 안 됩니다.
- **대응 방안**: Parser ➡️ Editor ➡️ Reviewer ➡️ Writer의 4단계 워크플로우가 모두 완료되어 렌더링 에러(Mermaid 오류, 찌꺼기 텍스트 등)가 완전히 0%가 된 무결점 상태일 때만 최종 완료를 선언해야 합니다.

## 3. 원시 덤프 데이터 방치 금지 (No Raw Dump Left Behind)
- **절대 규칙**: 파서가 1차 추출한 `> [!info] 🧩 **[도식 공간 분석]**` 등 시각 레이아웃 원시 데이터가 렌더링되지 않은 채 문서 내에 그대로 방치되는 것을 절대 금지합니다.
- **대응 방안**: Reviewer 에이전트 단계에서 문서 내에 원시 덤프 데이터(도식 공간 분석 등)가 단 1개라도 남아있는지 꼼꼼하게 전수 검사(QC)해야 하며, 발견 즉시 표(Table), 중첩 리스트(Nested List), 혹은 다이어그램(Mermaid) 중 가장 적합한 마크다운 형태로 100% 누락 없이 렌더링(구조화)해야 합니다.
